# Healthcare Provider Risk — an agent that flags providers before they get excluded

## The problem

In US healthcare, the OIG maintains a public list (the LEIE) of providers who have been excluded, meaning it's illegal for federal programs to pay them. A health plan that pays a claim to an excluded provider can end up liable for that money. Most of this checking happens reactively: providers get reviewed roughly in the order they come in, so low-risk and high-risk cases get the same attention, and the risky ones often surface late.

I wanted to build something that flips that around: score providers up front so the risky ones get looked at first, and let a non-technical reviewer just ask questions about the exclusion data in plain English.

## What I built

An agent that sits in front of two tools and decides which one a question needs:

1. **A risk-scoring tool** — give it a provider's NPI, it looks the provider up and returns a risk score from a trained model.
2. **A retrieval tool** — ask a general question about the exclusion data, it pulls the relevant records and answers from them, grounded in what it actually retrieved.

The agent reads the incoming question, works out the intent, and routes it. "What's the risk score for NPI 1871596098?" goes to the model. "Are there excluded providers in Texas?" goes to retrieval. The user doesn't have to know which is which.

## How it works

- **Data:** two public US federal datasets, both free to download — NPPES (the registry of every provider and their NPI) and the OIG LEIE (the excluded-provider list). I used a ~500K sample of NPPES cross-referenced against the full LEIE to build a labelled dataset.
- **The model:** an XGBoost classifier. The data is very imbalanced (excluded providers are rare), so plain accuracy is misleading and catching the rare positives (recall) is what matters. It outputs a 0–1 risk score used to sort a review queue.
- **The retrieval side:** the exclusion records are embedded with a sentence-transformers model and stored in a Qdrant vector database. A question retrieves the closest records, and the answer is generated by Gemini, grounded only in what was retrieved rather than the model's own memory. If the records don't support an answer, it says so instead of guessing.
- **The agent:** built with LangGraph as a small state graph — an LLM router node reads the question, decides the tool, calls it, and returns. This is the piece that makes it feel like one assistant instead of two separate scripts.

## Course concepts I used

- **Agents / routing:** an LLM decides which tool a request needs and calls it (LangGraph).
- **Retrieval-Augmented Generation:** answers come from retrieved documents, not model memory.
- **Embeddings + a vector database:** sentence-transformers into Qdrant for semantic search.
- **Grounding / controlled generation:** answers are constrained to the retrieved context and refuse when unsupported.
- **Combining classical ML with an LLM agent:** the XGBoost model is exposed as a tool the agent can call — the agent orchestrates both a predictive model and a language model.

## What's actually deployed vs. a prototype

I want to be straight about this because it matters:

- The **XGBoost scoring service is deployed** to Google Cloud Run (FastAPI + Docker).
- The **RAG pipeline and the LangGraph agent run in the notebooks** as a working prototype. They aren't deployed as a live service.

## What I'd add next

- A proper evaluation harness (RAGAS) with a golden question set, to measure the retrieval answers instead of eyeballing them.
- Role-based access on retrieval, so different users only see what they're allowed to.

## A note on responsible use

This is a portfolio and learning project built on public data. It's meant to *prioritise* human review, not to make a final decision about any provider. Anything real would need proper validation and a human in the loop.
